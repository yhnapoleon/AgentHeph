Guide Chatbot · 综合架构

面向内部系统的「讲解型」AI 助手：按权限解释功能、说明字段、指引导航；回答只基于知识库、绝不编造。本页汇总全部 guide 相关设计（含 KB 构建与 UI 摄取）。

① 三种 chatbot 类型 —— 由用户在前端选择

同一套平台支持三种能力档位，在 Studio 前端由接入者勾选（本质是 manifest 的 capabilities 字段）。选哪种，决定了读哪些源、装哪些工具、是否挂写路径。

类型能做什么主源是否需要
Guide（本页）讲解功能、说明字段、指引导航前端 UI（次源 swagger）只读知识；行级权限可选
Assistant（只读）查询/分析平台数据swagger / 数据 API + 代码只读数据工具 + per-row 行级权限
Edit在人工确认后改数据/配置同上 + 写接口写双门 + 草案确认 + executor 复核

关键洞察：三者主源不同——Guide 读 UI（用户能做什么、在哪做），Assistant/Edit 读 API（有哪些数据操作）。所以"选类型"不只是开关功能，它直接决定 KB 的构建方式。

② Guide 能做什么

  • 功能讲解：这个功能怎么用 / 这个页面做什么 —— 基于真实页面与字段。
  • 字段与枚举释义：这个字段什么意思 / 能填什么 / 这个状态值代表什么。
  • 导航指引：某功能在哪里 —— 直接给跳转链接带用户过去。
  • 只读数据查询（可选）：需要时查数据，按用户权限过滤可见范围。
  • 权限感知 + 诚实作答：只讲用户有权限看的；没有记录就说"未记录"，不臆造步骤。

边界：只读、只讲解，不改任何数据、不执行任何变更。

③ KB 是 Guide 的核心（双读者 + 三类知识）

Guide 的回答几乎都来自知识库（KB），所以 KB 构建是重中之重。要点：

  • 一卡两读者：每张卡的 body_md 给用户讲解；卡的 related_tools + 字段/enum 语义给 agent 用来选工具/取数。一次构建，两处复用。
  • 三类知识分开：①功能/UI 知识（怎么用，归卡片）②实时业务数据（具体值，归数据工具，绝不进卡片）③能力/路由（调哪个工具）。Guide 主攻①。
  • 分区结构化卡片，不做内容向量 RAG：按业务实体切成精确小卡，用 tool-calling 按需精确取；可审计、可溯源、省 token。

④ 源：UI 优先，swagger 降为增强（核心修正）

swagger 告诉你"有哪些数据操作"，UI 才告诉你"用户能做什么、在哪做、字段填什么"——后者才是 guide 要教的。所以 guide 的主源是前端 UI；swagger/代码降为次源，仅用于增强字段的取值枚举与校验规则。

UI 知识构成回答什么问题
导航结构（路由/菜单/tab）"X 在哪" + 生成跳转
可交互组件（按钮/表单/筛选/开关/表格/弹窗）"这里能做什么"
字段语义（label/placeholder/tooltip/帮助/校验提示）"这个字段什么意思/能填什么"
流程（多步向导、点完之后发生什么）"怎么做 X"
条件/权限（按钮何时禁用、哪个角色可见）权限可见性 + "你没权限做 X"

⑤ 怎么读 UI —— 三层叠加（结构 → 真相 → 描述）

层方法拿到什么性质
A 静态解析（主干）tree-sitter/babel 解析前端源路由表(页面/tab/deeplink)、组件→label/onClick、表单字段(zod/yup/RHF schema)、i18n 文案(用户可见词汇金矿)确定、便宜、稳定 ID
B 运行时爬取（地面真相）Playwright 抓无障碍树+DOM真实渲染的可交互元素(role=button/textbox/tab…框架无关)、可见文本、testid、截图、真实 URL抓条件渲染真相；复用 webapp-testing skill
C 视觉理解（补描述）截图喂多模态模型"这页做什么/这控件干嘛"的自然语言描述补 A/B；复用 BAU vision 模型

推荐组合 A+B+C：A 给结构骨架与稳定主键，B 给真实可交互元素与 deeplink（无障碍树是被低估的关键——它本就是语义化界面地图），C 补描述。每个抽取元素都带 provenance（路由/组件 id/testid），供卡片溯源与 UI-diff。

⑥ 功能聚类策略怎么生成

  1. 用导航树做种子（确定性主干）：顶层路由/菜单天然是候选分区——UI 导航本就是产品设计好的人类聚类，直接拿。
  2. LLM 精炼：喂 导航树 + 组件清单 + 实体模型，产出按 domain/entity 归并的聚类（合并同实体页面、拆分过大页面）。
  3. 主键去 UI 化：area 用领域名（如 issue_management），UI 页面只作 source_ref——tab 改名不塌分区。
  4. 人工审核：聚类决定整个 KB 结构与 app_knowledge 的 enum，必须人审冻结。
  5. eval 验证粒度：测"在哪/怎么用"是否路由到对的 area；准确率低或 area 过大就拆。粒度由 eval 定，不是固定阈值。

经验粒度：一个 area ≈ 一个顶层导航区/连贯任务域；area 内再按 kind（feature/howto/navigation/field）切卡。

⑩ 不同前端 → 不同读法

"怎么读 UI"高度依赖前端技术栈。摄取器应做成可插拔，按前端类型选策略：

前端类型静态解析(A)运行时(B)/视觉(C)
React/Vue SPA（如 BAU）解析 JSX/SFC + router + i18n + 表单 schemaPlaywright 无障碍树为主
Angular解析模板 + 路由模块 + i18n同上
服务端渲染（Jinja/Rails/Thymeleaf）解析模板文件 + 路由定义DOM 爬取为主
移动端 App解析布局/资源串(若有源)无障碍树 + 视觉为主
无源码 / 遗留系统—仅 Playwright 无障碍树 + 视觉（黑盒摄取）

设计含义：UI 摄取器是一组按前端类型可插拔的适配器，统一输出同一份"UI 清单"schema，后续聚类/生成/审核流程与前端无关。

⑪ 同步：spec-diff 变成 UI-diff

Guide 的源是 UI，所以同步机制也变：oasdiff 这类 swagger 差异工具不够用。改为对"UI 清单"（路由/组件/i18n/表单字段集合）打快照、做 diff：

  • 前端变更（路由新增 / 组件删除 / label 改名）→ UI 清单重算 → 与快照 diff；
  • 按 source_refs 反查受影响卡片 → 标记陈旧 → 回流人工审核重建。

机制与原 spec-diff 同构（快照 + hash + 反查），只是 diff 对象从 OpenAPI 换成 UI 清单。

⑫ 最该守的红线

写没了，但 guide 是散文密集的回答，模型最爱"按见过的同类系统编造功能/步骤"。所以红线集中在：
答案只能基于取到的卡片；卡片里没有的功能/步骤/字段一律说"系统未记录"，绝不用常识脑补一套看似合理的操作流程。这条进 eval 卡关（问一个不存在的功能，看它会不会编）。

一句话概括

Guide 助手 = 通用引擎只读运行 + KB 为核心；KB 的源是前端 UI（A静态/B爬取/C视觉，按前端类型可插拔）、swagger 仅增强；按导航树聚类成领域分区、LLM 起草 + 人工审核成卡片；运行时按权限取卡作答、只讲卡片有的、绝不编造。三种类型（guide/只读/edit）在前端由用户选，选 guide 就走这条 UI 优先的构建路径。
