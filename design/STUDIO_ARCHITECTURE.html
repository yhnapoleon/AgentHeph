Studio · 架构与功能介绍（控制平面）

Studio 是「造 chatbot 的工厂」——接入源、生成工具与知识、配权限、评测、发布。它产出配置，交给运行时去跑。

自建核心 复用成熟开源 LLM 生成器（借主模型） 人工审核环节 注册 / 存储

逐块说明

  • Studio 前端：造 bot 的操作台——配置向导（表单或对话式引导）、源接入、工具范围勾选、RBAC 编辑、卡片审核、spec-diff 审核、评测面板。
  • 控制面 API：管理项目/bot/部署，负责 Manifest 的增删改与版本化，并编排整条构建流水线。
  • 源接入：把外部输入拉进来——GitHub 代码（解析结构）、上传文件、swagger（解析并精简）、MCP server（自省出工具与 schema 摘要）。
  • 生成器（主 LLM 辅助）：用强模型把源转成产物——重塑工具、起草知识卡片、组装 system prompt、建议 RBAC、生成评测黄金集。只起草，不直接生效。
  • 人工审核（强制关卡）：所有生成产物（工具/卡片/prompt/判据/RBAC）必须人工批准并冻结后才可用——防止生成内容把错误当权威。
  • 评测引擎：用黄金集跑候选部署、判定 gate；管理 hold-out 集；接入对抗/红队工具。
  • 同步 / spec-diff：监听接口/代码变更 → oasdiff 出差异 → 按来源反查受影响的工具与卡片 → 回流到人工审核重建。
  • 发布：过了评测才能 promote 到生产；支持回滚。
  • 注册 / 存储：Studio 的产出落点，也是与 Runtime 的交接点——Manifest 注册表、部署注册表、卡片索引库、spec 快照、密钥库。
  • 基础设施：数据库、异步队列、对象存储、可观测，均为现成组件。

流程主线：接入 → LLM 生成 → 人工审核 → 评测 → 注册 → 发布，上线后由 spec-diff 闭环保持与源同步。每个生成产物都必须经人工审核与评测 gate 才能进入生产。

四、一句话概括

Studio 是控制平面：把外部源经「LLM 生成 + 人工审核 + 评测」变成一份可版本化的配置与知识库，注册后交给数据平面（Runtime）去跑。它造 bot，不跑 bot。

由设计讨论生成 · 整个 Studio 属 Post-MVP（路线图 P6），本图为目标架构 · 如需离线渲染，将末尾 mermaid 脚本替换为本地文件。
