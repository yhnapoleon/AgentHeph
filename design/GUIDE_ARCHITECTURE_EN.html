Guide Chatbot · Architecture

A read-only "explainer" assistant for an internal app: explains features, fields and navigation by user permission; answers strictly from the knowledge base, never fabricates. Includes the full UI→KB build pipeline.

1. Three bot types — selected by the user in the frontend

One platform, three capability tiers, chosen by the integrator in Studio (the manifest capabilities field). The choice decides which sources are read, which tools are loaded, and whether a write path exists.

TypeCan doPrimary sourceNeeds
Guide (this doc)Explain features, fields, navigationFrontend UI (OpenAPI secondary)Read-only knowledge; row-level RBAC optional
Assistant (read-only)Query / analyze platform dataOpenAPI / data API + codeRead-only data tools + per-row RBAC
EditChange data after human confirmationAbove + write endpointsWrite double-gate + proposal + executor recheck

Key insight: the primary source differs — Guide reads the UI (what a user can do and where), Assistant/Edit read the API (what data operations exist). So "pick a type" directly determines how the KB is built.

2. What Guide does

  • Feature explanation: how a feature works / what a page is for — grounded in real pages and fields.
  • Field & enum meaning: what a field means / what values are allowed / what a status value represents.
  • Navigation: where a feature lives — returns a deep link.
  • Read-only data lookup (optional): query data when needed, filtered by the user's permissions.
  • Permission-aware + honest: only shows what the user may see; says "not recorded" when unknown, never invents steps.

Boundary: read-only, explanation-only — never modifies data, never performs an action.

3. The KB is Guide's core

  • One card, two readers: a card's body_md explains to the user; its related_tools + field/enum semantics let the agent pick tools / build queries. Build once, reuse twice.
  • Three knowledge types kept apart: (1) feature/UI knowledge (how-to → cards) (2) live business data (values → data tools, never in cards) (3) capability/routing (which tool). Guide focuses on (1).
  • Partitioned structured cards, no content vector RAG: precise small cards by business entity, fetched on demand via tool-calling — auditable, traceable, token-efficient.

4. Source: UI-first, OpenAPI as enrichment

OpenAPI tells you "what data operations exist"; the UI tells you "what a user can do, where, what to type" — that is what Guide teaches. So the primary source is the frontend UI; OpenAPI/code is secondary, used to enrich field enums and validation and to cross-validate liveness.

5. User inputs & RBAC-first scan plan

Studio asks for three inputs:

  1. Frontend source — Git URL + branch/commit, or a source ZIP.
  2. Backend contract — OpenAPI URL, or an OpenAPI JSON/YAML file.
  3. Test environment — a test URL, the RBAC roles, and a test identity / auth profile per role.

RBAC is configured before scanning:

enter roles manually (default)  OR  extract role candidates from source/auth config (optional)
  -> user confirms
  -> bind a test identity per role
  -> freeze this Scan Plan

UI scanning only verifies role visibility; it can never create or modify RBAC, and never replaces the backend DataScopeAdapter.

7. Source extraction branch

ProjectDetector → ProjectProfile (React version, TS config, bundler, router, i18n, form lib, API client paths, e2e paths). Then:

ExtractorOutput factsTooling
RouteExtractorregistered routes, patterns, lazy imports, page component, route guard, required roles, feature flags, menu/breadcrumb, deeplinkts-morph
ComponentExtractorcomponent name, props, button/link/tab, headings, labels, tooltips, help text, dialog/drawer, table columns, data-testid, ARIA, disabled/hidden conditionsts-morph + react-docgen (props/docs only)
FormExtractorfield name, label, required, input type, min/max, regex, enum, defaults, help, validation msg, submit targetRHF / Zod adapters
I18nExtractortranslation key (kept, not just the string), languages, labels, tooltips, messagesi18next-parser / FormatJS
ApiClientExtractorfetch/axios/generated client, React Query hooks, component→API referencests-morph
KnipScannerdead-code signals only: unused export, orphan file, unresolved dynamic import, deprecated annotationknip

Live defaults, user data and API-returned business values must never enter the KB. Knip produces signals, it never deletes or excludes on its own.

8. OpenAPI branch

  • Load & canonicalize (swagger-parser): validate, resolve $ref, normalize to 3.x, canonical JSON, compute spec digest.
  • Extract capabilities: operationId, method/path, params, request/response schema, enums, constraints, description, security scheme, required scope, deprecated.
  • Align with frontend API client by: operationId → method+normalized path → generated client method → source URL pattern. A miss yields a conflict, never an automatic "frontend is dead".
  • Effect candidates only: infer a provisional effect.class (read_only / create / update / delete / external_send / unknown), reviewed:false. Inferred effects must be human-confirmed before they become security policy.

9. Runtime safe shallow scan (per role)

Run once per confirmed role. Auth precedence: existing Playwright auth fixture → test service account → one-time interactive login → store a TTL-bound encrypted auth_profile_ref. Never store credentials in the manifest. Seeds come from registered routes + menus + sitemap + e2e paths + manual entries.

Allowed (no side effects)Forbidden (side effects)
open route, click nav link, switch tab, open/close dialog, expand accordion, hover/focus tooltip, change side-effect-free filters, paginatesubmit form, Save, Delete, Approve, Send, Upload, pay, change config

Captured per UI state

Capture: URL, route pattern, title, ARIA tree, visible interactive elements, stable locators, tab/dialog/drawer state, screenshot, redirects, network request method/path, HTTP status, console errors.
Never capture: response bodies, Authorization header, cookies/tokens, user-entered values, live business table values.

InventoryNormalizer.merge(SourceFacts, ApiFacts, RuntimeFacts) also: turns /issues/123 into /issues/{issueId}, strips timestamps and dynamic counts, unifies i18n keys, merges multi-role results for one component, computes a stable content hash, keeps all provenance.

11. Evidence resolution (matrix, not an opaque score)

EvidenceResolver.resolve(...) assigns a status per item instead of a single confidence=0.73:

StatusCondition
verifiedexists in source AND reached at runtime by at least one role
supportedroute registered, corroborated by API/e2e
source_onlyfound only in source
runtime_onlyexists at runtime but source mapping failed
conflictthe three sources disagree
deprecated_candidatemultiple deprecation signals stack up
excludedexplicitly excluded by a human

Rules: runtime presence is strong positive evidence; runtime absence is only weak negative evidence. Knip alone cannot exclude. A missing OpenAPI endpoint alone cannot mark dead. No analytics hit alone cannot mark dead. Any RBAC conflict goes to a human.

12. Topic planning & five card kinds

Do not key knowledge on page/tab names. TopicPlanner.plan(UIInventory) keys on domain / entity / capability / topic — e.g. issue_management / issue / filtering / sla_risk_filter; UI pages are only source_refs.

Card kindAnswers
Featurewhat it does, who sees it, where the entry is
Navigationhow to get there, the path, required role
Fieldmeaning, required?, constraints
Enumallowed values and what each means
How-tohow to complete an action

How-to steps may use only evidenced transitions: source transitions, e2e steps, Playwright safe-scan steps, or existing help/onboarding copy. No evidence → no steps generated. The LLM reads normalized facts only, never the raw untrusted repo.

Card schema

{
  "area": "issue_management",
  "topic": "sla_risk_filter",
  "kind": "howto",
  "title": "Filter issues at SLA risk",
  "body_md": "Open Issues, expand Filters, enable 'SLA risk only', then apply.",
  "visible_to": ["viewer", "bau_member"],
  "related_tools": ["list_issues"],
  "source_refs": [
    {"type":"code","file":"src/issues/IssueFilters.tsx","symbol":"IssueFilters"},
    {"type":"runtime","scan_run":"run-123","role":"viewer","state":"filters_open"},
    {"type":"openapi","operation_id":"listIssues"}
  ],
  "evidence_status": "verified",
  "content_hash": "sha256:..."
}

Cards must not contain issue IDs, usernames, ticket contents or any live business value.

13. Tiered human review

Every card needs human approval, but the depth differs.

Batch approval (only if all hold)Per-card review (any triggers)
verified; no sensitive permission; no evidence conflict; every statement has source_refs; no write actionconflict / source_only / runtime_only / deprecated_candidate; role-visibility mismatch; uncertain effect; how-to with submit/modify steps; references a low-trust source

14. Eval gate

Deterministic checks (CardValidator)

schema valid; source_refs exist and resolve; no live data; no password/token; visibility not beyond evidence; related_tools exist; how-to steps map to transitions; no deprecated API referenced.

LLM checks (fuzzy only)

accurate feature summary; clear; introduces no step absent from evidence; does not state "maybe" as "certain".

Guide-bot regression (golden, human-confirmed)

e.g. "How do I filter SLA-risk issues?", "Can a viewer access the runbook?", "Where is the nonexistent bulk-delete feature?". Deployment gates: no fabrication, scope respected, correct knowledge routing, nonexistent feature rejected, no live values in KB. Failing the gate blocks release.

15. Freeze, release & incremental update

Freeze on release: repo commit, OpenAPI digest, RBAC scan plan, runtime scan digest, UI inventory digest, card versions, source_refs, eval result, reviewer. Production runtime reads only the immutable KBRelease, never unreviewed Studio drafts.

Incremental update on change: repo commit changed → rerun SourceExtractor; OpenAPI digest changed → rerun ApiExtractor + oasdiff; UI/runtime snapshot changed → rerun affected roles/routes. ImpactAnalyzer.diff(prev, new) → affected fact IDs → affected card IDs → regenerate only those → review → eval → new immutable release. Production KB is never edited in place; always a new version, always rollback-able.

In one line: Guide = read-only engine + KB as the core; the KB is built from UI-first sources (source code, OpenAPI, safe runtime scan), triangulated via an evidence matrix, clustered into domain topics, drafted by a schema-constrained LLM, human-reviewed in tiers, eval-gated, and shipped as an immutable release the runtime consumes read-only.
